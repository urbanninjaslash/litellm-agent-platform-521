HTML plan

Agent templates become user-owned JSON specs with an internal library and a global publish path.

This plan keeps LAP templates as JSON, lets internal users create and reuse their own templates from /agents, and adds a "Publish globally" action that opens a PR against the LAP template JSON.

Current state Built-in templates are fixed in src/agent_templates.json and rendered on /agents/new.
Target state Users see all internal templates, create new JSON templates, then create agents from them.
Global publish A template owner can generate a PR that adds the template JSON to the LAP global catalog.
Discover

Templates move to the agents surface

/agents gains a Templates tab next to the existing agent list. This is where internal users browse private, team, and global templates before creating agents.

Define

Template builder writes JSON

Users fill a form that mirrors the agent spec. A live JSON preview is always visible and is the canonical artifact saved by the platform.

Publish

Private first, global by PR

New templates start as internal templates. The global publish button quick-creates a PR that adds the JSON to LAP's checked-in template catalog.

Recommended JSON contract

Make the agent template spec close to the agent spec

The user-facing template JSON should use system instead of prompt, harness instead of harness_id, object-shaped tools, and a skills array. The backend can normalize this into the current create-agent API.

Template JSON Create agent API
system prompt
harness harness_id
skills skill_ids or generated skill files
tools tools plus selected MCP allowed tools
Implementation shape

Internal templates are JSON files with visibility metadata

  • Global templates remain checked into LAP and loaded from the repository.
  • Internal templates are saved as JSON documents owned by a user or team.
  • The Templates tab lists internal templates first, then global templates.
  • Every template has visibility: draft, internal, or global.
  • Every created agent stores template_id and template_version for drift detection.

Publish globally journey

1. Internal template works

User has already created agents from the template and knows it is useful.

2. Click Publish globally

LAP opens a publish drawer with JSON diff, validation status, and repo destination.

3. Validate against spec

Required fields, harness availability, model format, MCP IDs, and skills are checked before PR creation.

4. Create PR

Backend writes the JSON into the global template catalog and opens a branch plus PR.

5. PR merged

After merge and deploy, the template appears with a global badge for every LAP user.

What the user sees

  • Template name, owner, current internal version, and target global slug.
  • Validation checklist: JSON schema, harness exists, model present, skills resolvable, no private env var values.
  • Preview of the exact file that will be added to the LAP repo.
  • PR title and body, editable before creation.
  • After creation: PR link, branch name, and publish status on the template card.

MVP

What to build first

Add Templates tab, JSON schema, internal template create/edit, create-agent-from-template, and global PR creation using the normalized spec.

Guardrails

What to block

Do not publish templates with raw secret values, unknown harnesses, invalid MCP servers, invalid skill references, or JSON that cannot create an agent.

Success state

What good looks like

Internal users can define, browse, duplicate, use, and publish templates without editing source files by hand, while global templates still land as reviewed LAP JSON.
